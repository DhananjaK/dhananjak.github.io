Configure Egress IP to a Namespace in OpenShift 4 with the OVN Kubernetes CNI plugin

Overview#

OpenShift’s Egress IPs feature allows you to assign an IP address to a namespace or pod. In this way, all outbound traffic from that namespace or pod appears to originate from that IP address. In networking terms, it is known as a NATed IP address.

In this article, I’ll explain how to configure the Egress IPs feature in the OpenShfit project (Namespace). This mechanism is commonly used for security controls in enterprise-grade clusters.

Image Source: docs.openshift.com

Setup a Apache Webserver#

I installed a simple Apache web server as an external source on another host to validate the Egress IP service configuration. You can simply install the Apache web server on a Linux host as below.

sudo yum install httpd -y

sudo systemctl start httpd

sudo  systemctl enable httpd

echo "Test Egress Traffic" > /var/www/html/index.html

Create namespaces#

For demonstration purposes, I’ll create two namespaces (Projects) as below.

oc new-project "with-egress-ip" --display-name "With Egress IP Namespace"

oc new-project "without-egress-ip" --display-name "Without Egress IP Namespace"

Deploy a Sample Application#

In this example, I have deployed a sample NGINX pod in both projects.

oc project with-egress-ip
oc create deployment test-nginx --image quay.io/redhattraining/hello-world-nginx

oc project without-egress-ip
oc create deployment test-nginx --image quay.io/redhattraining/hello-world-nginx

Now, if you notice the IP address of the source, the source IP appears as the NODE_IP of the worker03. By default, outbound traffic is NATed to the node’s IP address. If we move the workload to another node, we would receive the source IP as another node IP in the OCP cluster.

This approach is not compatible with traditional firewall systems in an enterprise network to identify and control workload requests/egress traffic. Then egress IP comes into play. Now let’s see how to configure the egress IP Configure Egress IP to a Namespace in OpenShift.

Egress IP with OVN Kubernetes SDN#

Now, we will switch to the with-egress-ip namespace and create the egress IP address for the with-egress-ip namespace as below.

vi egressip.yml

You can apply one or more egress IP addresses to the namespace. In my case, I only added one IP address to the namespace for demo purposes.

apiVersion: k8s.ovn.org/v1
kind: EgressIP
metadata:
  name: with-egress-ip
spec:
  # NOTE: these egress IPs are within the subnet range(s) in which my worker nodes
  #       are deployed.
  egressIPs:
    - 192.168.30.246    
  namespaceSelector:
    matchLabels:
      kubernetes.io/metadata.name: with-egress-ip

Then, we will create an egress IP address for the with-egress-ip namespace by applying egressip.yml.

oc apply -f egressip.yml

The next step is to verify that the allocated egress IP address is bound to the OCP cluster nodes.

[dhananjak@ansible-core ocp_project]$ oc get egressips
NAME             EGRESSIPS         ASSIGNED NODE                ASSIGNED EGRESSIPS
with-egress-ip   192.168.30.246   
[dhananjak@ansible-core ocp_project]$

According to the above output, the allocated IP address is not bound to the OCP cluster nodes. If it is not bound, we have to update the cluster nodes by labelling every node in the cluster.

Here, I have labelled all nodes in my cluster. Based on your requirement, you can select the specific nodes that workloads are running.

oc label nodes worker01.ocp.acme.lab k8s.ovn.org/egress-assignable=""
oc label nodes worker02.ocp.acme.lab k8s.ovn.org/egress-assignable=""
oc label nodes worker03.ocp.acme.lab k8s.ovn.org/egress-assignable=""

Now, you can verify the status by running the same command.

[dhananjak@ansible-core ocp_project]$ oc get egressips
NAME             EGRESSIPS         ASSIGNED NODE           ASSIGNED EGRESSIPS
with-egress-ip   192.168.30.246    worker03.ocp.acme.lab   192.168.30.246
[dhananjak@ansible-core ocp_project]$

Test Egress IP Traffic#

Now, we have successfully configured the Egress IP address in the namespace. Let’s test Egress IP traffic. As we did before, let’s try to execute the curl command for our external web server inside the pod running in the with-egress-ip namespace.

[dhananjak@ansible-core ocp_project]$ oc exec -ti -n with-egress-ip deploy/test-nginx -- curl http://192.168.30.108
<html>
<head/>
<body>Test Egress Traffic</body>
</html>

Now, let’s look at the Apache web server access.log file. If you notice, the source IP address has been changed to our egress IP address.

[root@web-01 /]# tail -f /var/log/httpd/access_log

192.168.30.246 - - [06/Feb/2024:13:07:35 +0300] "GET / HTTP/1.1" 200 56 "-" "curl/7.61.1"
192.168.30.246 - - [06/Feb/2024:13:07:36 +0300] "GET / HTTP/1.1" 200 56 "-" "curl/7.61.1"
192.168.30.246 - - [06/Feb/2024:13:07:37 +0300] "GET / HTTP/1.1" 200 56 "-" "curl/7.61.1"

If the workload is dispatched to a different worker or scaled to more replicas, then the source IP observed in our web server will also be the same Egress IP that we have configured in the respective namespace.
